# planktonzilla

## What This Is

`planktonzilla` is a Python deep-learning framework, model collection, and dataset toolkit for plankton image classification, built by Inria Chile as part of the OcéanIA project. It already exists as working internal research code (Hydra + HuggingFace `Trainer` pipeline, vendored `open_clip`, custom losses for class imbalance, dataset importers for the major plankton datasets). This milestone gets the project ready for **public use** — packaging existing trained models for external researchers, marine biologists, students, and paper/grant reviewers to consume.

## Core Value

**Curated, citable pre-trained plankton-classification models on Hugging Face Hub, each with a real model card, backed by a repo a stranger can credibly load a model from.**

If everything else slips, the HF Hub model release is the artifact that must ship.

### Out of Scope

<!-- Explicit boundaries with reasoning. -->

- **Pip-installable PyPI release** — deferred. The "publish models + working clone-and-load" path covers researchers and biologists; PyPI broadens distribution but is a separable effort. Add in a follow-up milestone once feedback comes in.
- **Saliency / Grad-CAM in the demo** — deferred to v1.1. Across CNN (ResNet/EfficientNet) and ViT/CLIP architectures the technique is non-uniform (Grad-CAM vs attention-rollout); shipping it correctly across all 7 models is more work than it's worth at v1.
- **Full install + CI hardening** — separate effort. The training-side bare `except:` ([planktonzilla/train.py:167](../planktonzilla/train.py#L167), [planktonzilla/clip_model.py:39](../planktonzilla/clip_model.py#L39)), deprecated `F.log_softmax(logits)` without `dim=` ([planktonzilla/loss.py:63](../planktonzilla/loss.py#L63)), and `skip_in_github_ci` ([tests/shared.py:19-25](../tests/shared.py#L19-L25)) all stay broken until a dedicated hardening milestone.
- **Re-vendoring or un-vendoring `open_clip` cleanly** — deferred. Pinning to upstream SHA / declaring in `pyproject.toml` only happens if the model-load path requires it; otherwise leave for hardening milestone.
- **New training campaign** — out of scope. The winning checkpoints are already identified; v1 is curation + packaging, not new runs.
- **New datasets, new model architectures, new losses** — out of scope. Ship what exists, then iterate.
- **Pretty docs site (MkDocs / Sphinx / GitHub Pages)** — deferred. README is the documentation surface for v1.
- **Per-dataset benchmark paper / leaderboard** — out of scope. Models ship with eval numbers in their cards; a unified benchmark write-up is a separate research artifact.

## Context

**Existing codebase:** Mature internal research project. See `.planning/codebase/` for the full map (`STACK.md`, `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `INTEGRATIONS.md`, `CONCERNS.md`).

**Architecture in one paragraph:** Hydra-composed configs under [configs/](../configs/) drive two `@hydra.main` entry points (`pz_train`, `pz_import_dataset`) defined in [planktonzilla/train.py](../planktonzilla/train.py) and [planktonzilla/dataset_import/import_dataset.py](../planktonzilla/dataset_import/import_dataset.py). Training instantiates a HuggingFace `AutoModelForImageClassification` (or `ClipClassifier` wrapping a vendored `open_clip` visual tower) and runs through `transformers.Trainer` with optional PEFT/LoRA, custom imbalance loss, augmentations, and W&B/MLflow/Trackio tracking. Dataset imports go to `huggingface.co/project-oceania`.

**Audiences (all matter, different needs):**
- ML researchers — care about reproducibility, configs, ablations, paper citations.
- Marine biologists / domain experts — care about install simplicity, pre-trained models, the demo, model-card "intended use" sections.
- Students / new contributors — care about README clarity, working examples, easy first PRs.
- Paper / grant / funder reviewers — care about a frozen, citable artifact with real eval numbers.

The single artifact that serves the most audiences at once is the curated HF Hub release — hence the Core Value framing.

**Known issues that the codebase map flagged but are explicitly out of scope for v1:** see CONCERNS.md High-severity items (broken `transformers` constraint resolved by `poetry.lock`, vendored `open_clip` divergence, `skip_in_github_ci`, bare excepts, deprecated `F.log_softmax` calls). HARD-01 above only covers the subset that blocks the public-facing "load and use a model" path.

## Constraints

- **Tech stack**: Python 3.11+, Poetry, HuggingFace ecosystem (`transformers`, `datasets`, `huggingface_hub`), Hydra 1.3, PyTorch 2.x, vendored `open_clip` — locked by existing codebase. New work must integrate, not replace.
- **Distribution**: HF Hub (`huggingface.co/project-oceania`) is the model registry. GitHub (`Inria-Chile/deep_plankton`) is the code/issue surface. HF Spaces is the demo host. No PyPI in v1.
- **Compatibility**: The "load a published model" snippet must run on a clean Python environment (no clone of the planktonzilla repo required). This constrains how `ClipClassifier` and any custom loss tying must be packaged.
- **Quality bar**: "Ship when it's right" — no external deadline. Each shipped artifact must be defensible to a paper reviewer.
- **License**: MIT for code (already set). Each HF model card declares its own license, which may inherit constraints from the source dataset's license.
- **Authority**: User has full institutional authority to publish. No Inria legal / dataset-license / co-author gate to add.
- **Out-of-scope discipline**: The codebase has many tempting-to-fix issues (CONCERNS.md). v1 ignores them unless they block the public-facing artifacts. This is the primary discipline this milestone is asking for.
